## 1. Descripción del Cambio

Actualizar `CLAUDE.md §6 — Flujos de Trabajo` para documentar cinco decisiones arquitectónicas tomadas en sesión del 21 de marzo de 2026 que no quedaron registradas en la constitución del proyecto:

1. **Rol de `main`:** La rama `main` permanece como rama de gobernanza permanente (documentación, SDD, resúmenes ejecutivos). La rama `prod` es la rama de código en producción. No son equivalentes ni sinónimas.
2. **CI Quality Gate (GitHub Action):** Action que corre automáticamente en cada push/PR. Ejecuta dos jobs en paralelo: (a) tests del engine Python con `pytest` + reporte JSON, (b) tests del dashboard con `npm test`. Requiere secrets `SUPABASE_URL`, `SUPABASE_KEY`, `SUPABASE_SERVICE_ROLE_KEY`. Adaptaciones para Demo_Bunuelos: carpeta dashboard es `web/` (no `dashboard/`), tests del engine en `engine/tests/`, escucha en PRs hacia `dev`/`test`/`prod` (no hacia `main`). Se implementa en Etapa 2.1.
3. **Code Review obligatorio:** El skill `/code-review-demo` (a crear en Etapa 2.1) se ejecuta automáticamente en cada PR hacia `dev`. Es un gate: si falla, no se permite el merge.
4. **Versionamiento semántico:** GitHub Action `release-please` genera versiones automáticas (MAJOR.MINOR.PATCH) al hacer merge a `prod`. Se implementa en Etapa 2.1.
5. **Clarificación del Paso Crítico:** La sincronización `test → dev` ocurre **antes** del merge final a `prod`, no después. Orden explícito: `test → dev (sync) → prod`.

## 3. Alcance e Implicaciones

Este cambio es **puramente documental** — no modifica código, configuración ni estructura de carpetas. Solo actualiza texto en CLAUDE.md.

### Riesgos de NO hacer el cambio:
- Un agente nuevo en Etapa 2.1 no sabrá cuándo ejecutar code review ni cómo gestionar las ramas
- La ambigüedad del "Paso Crítico" en §6 puede llevar a sincronizar `test → dev` en el orden incorrecto
- `release-please` podría implementarse sin documentación oficial → deuda técnica silenciosa
- El CI Quality Gate podría implementarse con rutas incorrectas (`dashboard/` en lugar de `web/`, o escuchando en `main` en lugar de `dev`/`test`/`prod`)

### Riesgos de SÍ hacer el cambio:
- Ninguno relevante — es documentación que refleja decisiones ya tomadas y aprobadas por el usuario
