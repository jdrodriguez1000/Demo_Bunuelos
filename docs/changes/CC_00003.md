## 1. Descripción del Cambio

Agregar a `CLAUDE.md §5 — Estándares y Convenciones de Código` una regla constitucional de **TDD (Test Driven Development) obligatorio** para todos los archivos Python creados en cualquier etapa del proyecto.

Decisiones específicas tomadas en sesión 2026-03-22:
1. **TDD obligatorio** para todos los módulos Python, sin excepción de etapa o tipo de archivo.
2. **Integration tests** como estándar para conectores e infraestructura (prueba contra servicios reales, no mocks).
3. Los tests viven en `engine/tests/` con la misma estructura de subcarpetas que `engine/src/` (ej. `engine/tests/connectors/` espeja `engine/src/connectors/`).
4. **`engine/tests/` se anticipa junto con cada módulo nuevo**, igual que se anticipó `engine/src/connectors/` en F01_E02.
5. Los tests de integración de conectores quedan incluidos en el **CI Quality Gate** (Etapa 2.1).

## 3. Alcance e Implicaciones

Este cambio es **puramente documental** — agrega una regla a `CLAUDE.md §5`. No modifica código existente.

### Impacto inmediato (F01_E02):
- `engine/tests/connectors/` se añade al alcance de F01_E02 (junto con `engine/src/connectors/`)
- El PRD `f01_02` debe actualizarse con REQ-09

### Impacto estructural (todas las etapas futuras):
- Cada módulo nuevo en `engine/src/` requiere su test paralelo en `engine/tests/`
- Ningún archivo Python puede considerarse "hecho" sin su test de integración correspondiente

### Riesgos de NO hacer el cambio:
- Módulos sin tests llegan a Fase 2 sin cobertura → deuda técnica acumulada
- El CI Quality Gate de Etapa 2.1 no tiene tests que ejecutar → gate vacío sin valor
- Regresiones silenciosas al modificar conectores en etapas avanzadas

### Riesgos de SÍ hacer el cambio:
- Leve aumento de tiempo por etapa (escribir test antes del código)
- Mitigado: los integration tests de conexión son simples y reutilizables
