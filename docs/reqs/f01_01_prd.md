# PRD — Definiciones y Cimientos: Gobernanza y Estructura Base (`f01_01`)

> **Etapa:** 1.1 — Reglas globales, estructura de archivos e inicialización del `PROJECT_index.md`
> **Fase:** 1 — Definiciones y Cimientos
> **Estado:** 🟡 En Progreso
> **Última actualización:** 2026-03-20

---

## 1. Resumen Ejecutivo

Esta etapa establece los cimientos no negociables del proyecto. Su propósito es garantizar que el agente IA opere bajo reglas claras, que el repositorio tenga una estructura coherente y que existan los mecanismos de contexto necesarios para retomar el trabajo con precisión entre sesiones.

Sin esta etapa completada, cualquier trabajo posterior carece de gobernanza, trazabilidad y reproducibilidad. Es el prerequisito absoluto de todas las fases siguientes.

---

## 2. Objetivos de Negocio

- `[OBJ-01]` Establecer la constitución del proyecto (`CLAUDE.md`) como fuente de verdad única para el comportamiento del agente IA, garantizando consistencia entre sesiones.
- `[OBJ-02]` Definir y crear la estructura de carpetas base del repositorio para que el proyecto escale sin deuda técnica de organización.
- `[OBJ-03]` Inicializar los mecanismos de tracking de contexto (`PROJECT_index.md`, `PROJECT_handoff.md`) que permiten retomar el trabajo sin fricción ni pérdida de estado.
- `[OBJ-04]` Crear las habilidades de Claude Code (`/update-index`, `/session-close`, `/sdd-doc`) que automatizan la gobernanza del proyecto.
- `[OBJ-05]` Preparar el entorno de desarrollo Python (`venv`) con las dependencias base del motor de IA.
- `[OBJ-06]` Establecer el mecanismo de registro de lecciones aprendidas para capturar conocimiento operativo por sesión y etapa.
- `[OBJ-07]` Crear el mecanismo de resumen ejecutivo al cierre de cada etapa, en lenguaje de negocio, como gate de avance.
- `[OBJ-08]` Implementar el sistema de Control de Cambios (CC) para garantizar trazabilidad ante modificaciones no contempladas o en etapas cerradas.

## 7. Riesgos y Supuestos

| ID | Descripción | Probabilidad | Mitigación |
|---|---|---|---|
| `[RSK-01]` | `CLAUDE.md` incompleto o ambiguo genera comportamiento impredecible del agente en etapas posteriores. | Alta | Revisión exhaustiva del documento antes de marcar la etapa como completada. |
| `[RSK-02]` | Skills de Claude Code mal configurados (frontmatter incorrecto o triggers erróneos) interrumpen el flujo de gobernanza. | Media | Prueba de invocación de cada skill al final de la etapa como parte del DoD. |
| `[RSK-03]` | Versión de Python incompatible con las dependencias del stack (`skforecast`, `lightgbm`). | Baja | Documentar versión exacta usada en `requirements.txt` con pins de versión (`==`). |
| `[RSK-04]` | `venv/` accidentalmente commiteado al repositorio por `.gitignore` mal configurado. | Media | Verificar `.gitignore` antes del primer commit. |
| `[RSK-05]` | Sin Resumen Ejecutivo, los stakeholders no técnicos pierden visibilidad del avance y no pueden tomar decisiones informadas sobre el proyecto. | Alta | El skill `/close-stage` es un gate obligatorio antes de avanzar de etapa. `CLAUDE.md` lo prohíbe explícitamente. |
| `[RSK-06]` | Sin Control de Cambios, los documentos SDD se desincronizan silenciosamente del código, invalidando su valor como fuente de verdad. | Alta | El Mandato de Control de Cambios en `CLAUDE.md` §1 obliga al agente a pausar y ejecutar `/change-control` ante cualquier cambio no contemplado. |
