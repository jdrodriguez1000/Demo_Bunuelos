## 1. Resumen Ejecutivo

Esta etapa crea el **Data Contract**: un archivo YAML máquina-legible (`contracts/data_contract.yaml`) que formaliza el acuerdo estructural entre Cafetería SAS (productor de datos) y el equipo de IA (consumidor de datos).

El contrato define **qué tablas existen, qué columnas tiene cada una, qué tipos de datos exactos se esperan, y cuáles tablas son mandatorias** para construir la variable objetivo del modelo de pronóstico: `demanda_teorica`.

**Diferencia fundamental con la etapa de validación:** Esta etapa construye el contrato (estructura). La etapa de validación posterior lo usa para verificar que los datos reales del cliente cumplen **exclusivamente lo estipulado en el contrato**: que las tablas existan, que las columnas tengan los nombres y tipos correctos. La validación de reglas de negocio, valores nulos, centinelas y calidad de datos se realiza en una etapa separada y posterior. Aquí no se valida nada — solo se define la estructura acordada.

**Artefacto central:** `contracts/data_contract.yaml` — archivo versionado, inmutable una vez publicado, persistido en la nube (Supabase + S3), y requisito obligatorio para que el Engine pueda iniciar.

## 8. Conceptos Clave del Data Contract

### 8.1 Contrato Dinámico y Extensible

El contrato NO es estático. Está diseñado para crecer con el proyecto. Actualmente incluye 7 tablas, pero en el futuro pueden agregarse más sin modificar el formato del archivo:

- **Agregar tablas opcionales:** Nueva versión del contrato. Solo requiere añadir una entrada al array `optional_tables`.
- **Agregar tablas mandatorias:** Nueva versión del contrato. Requiere que la nueva tabla participe en la construcción de la variable objetivo.
- **Eliminar tablas opcionales:** Nueva versión del contrato. Se remueve la entrada del array.
- **Eliminar tablas mandatorias:** **PROHIBIDO** — las tablas mandatorias son las que construyen la variable objetivo y no pueden ser removidas.

La estructura YAML usa arrays (`mandatory_tables: [...]`, `optional_tables: [...]`) precisamente para soportar N tablas sin límite ni cambio de formato.

### 8.2 Tabla Mandatoria

Una tabla es mandatoria si cumple **cualquiera** de estas condiciones:
1. Contiene la variable objetivo o un componente directo de ella.
2. Es necesaria para calcular/inferir la variable objetivo.

En el estado actual del proyecto:
- `usr_ventas` → Mandatoria (contiene `unidades_vendidas`, input de ambas reglas).
- `usr_produccion` → Mandatoria (contiene `unidades_sobrantes`, determina qué regla aplicar).

### 8.3 Variable Objetivo: `demanda_teorica`

La `demanda_teorica` es una variable **calculada** que representa la demanda real del mercado, corrigiendo el sesgo de agotamiento de inventario:

- **Regla 1 — Con sobrantes:** Si `unidades_sobrantes > 0`, no hubo restricción de inventario. `demanda_teorica = unidades_vendidas`.
- **Regla 2 — Sin sobrantes (agotado):** Si `unidades_sobrantes = 0`, el producto se acabó antes del cierre. Se infiere la demanda no satisfecha:
  1. Extraer `hora_ultima_venta` del día (de `usr_ventas.fecha` timestamp).
  2. Buscar días similares: mismo día de semana, últimos 90 días, excluyendo festivos, que SÍ tuvieron sobrantes > 0.
  3. Calcular promedio de ventas desde `hora_ultima_venta` hasta el cierre en esos días similares.
  4. `demanda_teorica = unidades_vendidas + demanda_no_satisfecha_estimada`.
  5. Fallback: si no hay al menos 1 día similar → `demanda_teorica = unidades_vendidas`.

### 8.4 Inmutabilidad y Versionamiento

- Cada contrato publicado es **inmutable**. No se modifica — se crea uno nuevo.
- Cualquier cambio, por mínimo que sea (incluso una descripción), genera una nueva versión.
- Solo **un contrato activo** por cliente (`pyme_id`) en cualquier momento.
- Al activar una nueva versión, la anterior se desactiva automáticamente (trigger en Supabase).

### 8.5 Persistencia Híbrida

- **Supabase (tabla `contracts`):** Metadatos, auditoría, referencia al archivo S3, checksum SHA256, flag `is_active`.
- **S3 (vía DVC):** Archivo YAML versionado como artefacto. El Engine lo descarga en startup.
- **Git (local):** `contracts/data_contract.yaml` se versiona en Git para gobernanza y revisión de código.

### 8.6 Error Bloqueador de Startup

Si el Engine no encuentra un contrato activo en Supabase/S3, **NO PUEDE INICIAR**. Debe emitir un error claro:
```
FATAL: No active data contract found for pyme_id '001_ABC'.
Action: Create and publish a contract using Etapa 1.3 process.
System cannot start without a valid data contract.
```
