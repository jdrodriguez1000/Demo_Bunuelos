> **Fecha:** 2026-03-22

---

## 1. Resumen Ejecutivo

Esta etapa establece la **conexión verificada y documentada** entre los dos consumidores del sistema (Engine Python y Dashboard Next.js) y la infraestructura cloud del proyecto (Supabase + S3). No se crean tablas nuevas ni se ingieren datos; se documenta la realidad existente y se valida que cada credencial funciona correctamente.

Al cerrar esta etapa, el equipo tendrá certeza técnica de que:
1. El Engine Python puede leer y escribir en Supabase usando la service role key.
2. El Dashboard Next.js puede leer datos de Supabase usando la anon key (con RLS activa).
3. El MCP de Supabase está configurado en Claude Code para auditoría de BD durante el desarrollo.
4. El acceso a S3 está documentado para el flujo DVC de modelos entrenados.
5. Existe el módulo de conexión Python en `engine/src/connectors/` como base para Fase 2.

> **Decisión Arquitectónica (2026-03-22):** Solo se crea la subcarpeta `engine/src/connectors/` en esta etapa. El resto de la estructura de `engine/` (pipelines, tests, main.py) se crea en Fase 2 según lo definido en CLAUDE.md §4.

## 7. Riesgos y Supuestos

| ID | Descripción | Probabilidad | Mitigación |
|---|---|---|---|
| `[RSK-01]` | Límite de 1.000 registros/query de Supabase puede truncar cargas históricas en Fase 2. | Alta | Documentar estrategia de paginación (cursor o range) en SPEC. Implementar en Etapa 2.2. |
| `[RSK-02]` | Credenciales en `.env` pueden expirar o ser revocadas. Precedente: token GitHub expuesto en Etapa 1.1 (lessons-learned). | Media | Verificar vigencia de claves antes de iniciar. Documentar proceso de rotación en SPEC. |
| `[RSK-03]` | Políticas RLS mal configuradas en Supabase podrían bloquear la anon key desde el Dashboard o permitir acceso no autorizado. | Media | Documentar y validar cada política con una query de prueba desde anon key (sin service role). |
| `[RSK-04]` | MCP Supabase puede requerir permisos adicionales en Supabase (ej. acceso a `information_schema`). | Baja | Validar en el paso de configuración; si falla, documentar restricciones. |
| `[RSK-05]` | Credenciales S3 en `.env` no validadas hasta Fase 2 — podrían estar desactualizadas. | Baja | Hacer un ping a S3 (list bucket) como parte de la validación de esta etapa. |
